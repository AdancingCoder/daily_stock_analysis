# 项目架构文档

> 本文档帮助项目新手快速理解整体架构、模块职责和开发流程。

## 一、项目概述

**股票智能分析系统（DSA）** 是一个基于 AI 大模型的 A股/港股/美股 自选股智能分析系统。核心流程：

```
抓取行情数据 → 技术/基本面分析 → 新闻搜索 → LLM 智能分析 → 生成报告 → 多渠道推送
```

### 三种运行模式

| 模式 | 说明 | 入口 |
|------|------|------|
| CLI 单次运行 | 执行一次完整分析 | `python main.py` |
| 定时任务 | 每日固定时间自动执行 | `python main.py --schedule` |
| Web 服务 | 提供 Web UI + REST API | `python main.py --webui` |

## 二、技术栈

| 层级 | 技术 |

## 四、核心模块详解

### 4.1 主流程 Pipeline (`src/core/pipeline.py`)

分析系统的核心编排器，负责串联整个分析流程：

```
1. 数据获取 (data_provider) → 2. 技术分析 → 3. 新闻搜索 (search_service)
→ 4. LLM 分析 (analyzer) → 5. 报告生成 → 6. 通知推送 (notifier)
```

关键特性：
- **线程池并发**：多只股票并行分析
- **容错隔离**：单股失败不影响其他股票
- **交易日过滤**：非交易日自动跳过（可 `--force-run` 强制）
- **合并推送**：个股结果 + 大盘复盘合并为一份报告

### 4.2 数据源层 (`data_provider/`)

多数据源适配 + Fallback 机制：

```
         ┌─────────────┐
         │  调用方代码  │
         └──────┬──────┘
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
 AkShare    Tushare    YFinance   ...（10+ 数据源）
    │           │           │
    └───────────┼───────────┘
                ▼
        标准化输出 (base.py)
```

- **优先级**：按 `data_provider/__init__.py` 中定义的顺序依次尝试
- **Fallback**：前一数据源失败自动降级到下一个
- **标准化**：所有 fetcher 输出统一的数据结构
- **支持市场**：A股（6 个数据源）、港股（Longbridge）、美股（YFinance/AlphaVantage/Finnhub）

### 4.3 分析器 (`src/analyzer.py`)

负责构建 LLM Prompt 并解析分析结果：

- 收集技术指标（均线、MACD、筹码分布、资金流等）
- 收集新闻和舆情
- 构建结构化 Prompt 发送给 LLM
- 解析 LLM 返回的 JSON/Markdown 报告
- 输出：操作建议、评分、趋势预测、风险警报、催化因素

### 4.4 Agent 问股系统 (`src/agent/`)

多轮对话式问股功能，支持 15+ 种内置策略：

```
用户提问 → Orchestrator 编排 → 选择策略 → 调用 Tool 获取实时数据
→ LLM 分析 → 返回回答（支持多轮追问）
```

| 子模块 | 职责 |
|--------|------|
| `orchestrator.py` | 对话编排，管理会话状态 |
| `executor.py` | 执行分析，调用 LLM |
| `runner.py` | 运行器，管理工具调用 |
| `factory.py` | 工厂，创建 Agent 实例 |
| `tools/` | 工具集（行情查询、K 线查询、新闻搜索等） |
| `skills/` | 技能文件 |
| `strategies/` | 策略定义（对应 `strategies/` 目录的 YAML） |
| `llm_adapter.py` | LLM 适配层 |

## 六、配置体系

### 6.1 环境变量 (`.env`)

核心配置项分类：

| 类别 | 关键变量 | 说明 |
|------|----------|------|
| AI 模型 | `ANSPIRE_API_KEYS`, `OPENAI_API_KEY`, `GEMINI_API_KEY` | 至少配置一个 |
| 数据源 | `TUSHARE_TOKEN`, `AKSHARE_*` | 按需配置 |
| 搜索服务 | `SERPAPI_API_KEYS`, `TAVILY_API_KEYS` | 推荐配置 |
| 通知渠道 | `WECHAT_WEBHOOK_URL`, `FEISHU_WEBHOOK_URL` 等 | 至少配置一个 |
| 自选股 | `STOCK_LIST` | 必填，如 `600519,hk00700,AAPL` |
| 调度 | `SCHEDULE_TIME`, `SCHEDULE_ENABLED` | 定时任务 |
| 功能开关 | `AGENT_MODE`, `BACKTEST_ENABLED`, `MARKET_REVIEW_ENABLED` | 按需开启 |

### 6.2 配置加载顺序

```
.env 文件 → 进程环境变量 → Config 单例 (src/config.py)
                    ↓
        ConfigManager (src/core/config_manager.py)
                    ↓
            WebUI 可动态修改
```

### 6.3 WebUI 热更新

Web 界面修改配置后会写入 `.env` 文件，定时任务和下次 CLI 运行会自动读取最新值。

## 七、开发指南

### 7.1 新增数据源

1. 在 `data_provider/` 下创建 `xxx_fetcher.py`
2. 继承 `StockDataFetcher` 基类（`data_provider/base.py`）
3. 实现 `fetch_realtime()`, `fetch_kline()`, `fetch_technical_indicators()` 等方法
4. 在 `data_provider/__init__.py` 的优先级列表中注册

### 7.2 新增通知渠道

1. 在 `src/notification_sender/` 下创建 `xxx_sender.py`
2. 实现统一的发送接口
3. 在通知服务中注册

### 7.3 新增分析策略

1. 在 `strategies/` 下创建 `your_strategy.yaml`
2. 定义策略名称、描述、Prompt 模板、所需数据
3. 重启服务即可生效（无需改代码）

### 7.4 新增 API 端点

1. 在 `api/v1/endpoints/` 下创建新文件
2. 使用 FastAPI 定义路由
3. 在 `api/v1/router.py` 中注册路由
4. 在 `api/v1/schemas/` 中定义请求/响应模型

### 7.5 新增 Bot 命令

1. 在 `bot/commands/` 下创建 `xxx.py`
2. 继承 `BaseCommand`
3. 在命令注册处注册

### 7.7 开发规范

详见 [AGENTS.md](../../AGENTS.md)，核心原则：

- **目录边界**：后端放 `src/`、`data_provider/`、`api/`、`bot/`；前端在 `apps/dsa-web/`
- **不写死配置**：密钥、路径、模型名等均通过环境变量
- **复用优先**：优先复用现有模块和配置入口
- **最小改动**：只做与任务直接相关的改动
- **文档同步**：涉及用户可见能力变化时同步更新文档

## 九、关键设计模式

### 9.1 Fallback 机制

数据源和通知渠道均采用降级策略：
- 数据源：优先级高的失败自动尝试下一个
- 通知渠道：单一渠道失败不影响其他渠道
- 搜索服务：多搜索引擎依次降级

### 9.2 单例模式

`Config` 类采用单例，确保全局配置一致：
```python
from src.config import get_config
config = get_config()
```

### 9.3 懒加载

`main.py` 中的 `StockAnalysisPipeline` 使用描述符实现懒加载，避免导入时产生副作用。

### 9.4 线程安全

- 大盘复盘使用文件锁防止并发执行（`market_review_lock.py`）
- 分析线程池隔离，单股失败不影响整体
